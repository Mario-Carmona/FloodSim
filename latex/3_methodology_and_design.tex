\label{chapter:methodology_and_design}

\section{Proposed Analytical Framework}

To fill the gap between what we learn from modeling and what a Reactive Digital Twin really needs, this research suggests a new way of 
looking at things. The plan is not about simulating water but about creating a system that can handle a lot of changing data like what 
happens during a flash flood, such as the 2024 Valencia DANA.

The new approach is based on three ideas that make sure the system is strong and accurate:

\begin{enumerate}
    \item \textbf{Mathematical Formalization of Discrete Space}: Utilizing the $m:n-CA^k$ paradigm to represent complex, non-linear hydraulic interactions through discrete transition rules. This method is faster than ways of solving Shallow Water Equations.
    \item \textbf{Technological Decoupling (Hexagonal Architecture)}: Implementing a strict separation between the simulation core and external concerns (GIS data, MQTT messaging, logging). This ensures that the physical logic remains pure and adaptable to different hardware or data sources.
    \item \textbf{Distributed Telemetry and Interoperability}: Designing a communication layer that transforms the simulator from a standalone tool into a distributed node. By using MQTT, the system enables real-time synchronization between the computational engine and remote visualization or decision-support interfaces.
\end{enumerate}

By putting these ideas we can move away from the old way of doing simulations, batch-processing, where we do one thing at a time. 
Instead we can do everything at the time: get data process it and share the results all within a system that is connected and working 
together. That is what the Reactive Digital Twin is all about.


\subsection{The Multilayer Conceptualization ($m:n-CA^k$)}

The cornerstone of the simulation logic is the adoption of the Multilayer Cellular Automata ($m:n-CA^k$) formalism. Traditional 
Cellular Automata (CA) typically operate on a single state per cell, which is insufficient for representing the heterogeneous variables 
of a river basin. The $m:n-CA^k$ generalization allows for a multidimensional representation where the environment is discretized into 
a stack of $n$ computational layers.

\paragraph{Mathematical Formalization}

Formally, the simulation environment is defined as a specialized cellular space where each cell $c$ at a given coordinates $(i, j)$ is 
characterized by a state vector $S_{i,j}$. The system is represented by the tuple:

$$A = \langle R, E, V, N, \Psi \rangle$$

Where:

\begin{itemize}
    \item $R$ is the $m$-dimensional cell space.
    \item $E$ is the set of $n$ layers that define the global state and the physical environment of the system.
    \item $V$ represents the vicinity function.
    \item $N$ represents the nucleus function.
    \item $\Psi$ is the combination or evolution function that governs the fluid dynamics.
\end{itemize}


\subsubsection{Cell Space ($R$) and Layer Definition ($E$)}

The simulation space $R$ is defined on a regular two-dimensional grid $R \subset \mathbb{Z}^2$, where each cell is 
identified by its spatial coordinates $(i, j)$. The physical size of each cell is determined by the spatial resolution 
$\Delta x$.

The system consists of a set $E = \{E_1, E_2, E_3, E_4, E_5, E_6\}$ of $n=6$ computational layers. One of the fundamental characteristics 
of the $m:n-CA^k$ model is that the layers may differ in their geometric and temporal dimensions. To optimize memory layout and performance, 
transient layers (such as raw geospatial Land Cover) are purged post-initialization, leaving the system to operate on the following 
persistent tensor representations:

\begin{enumerate}
    \item $E_1$: Topobathymetry ($Z_{topo-bathy}$). A static 2D layer of dimension $X \times Y$. It combines the Digital 
    Elevation Model (DEM) with the bathymetry of the watercourses, providing the absolute elevation of the riverbed: 
    $E_1 \subset \mathbb{R}^{X \times Y}$.
    \item $E_2$: Surface Roughness ($\mu_{soil}$). A static 2D layer of dimension $X \times Y$. Although derived from discrete 
    Land Cover (LC) classifications via a State Transfer Function ($LC \xrightarrow{f} \mu_{soil}$), this continuous layer of 
    Manning's $n$ coefficients is the one that strictly persists during execution to govern horizontal friction: 
    $E_2 \subset \mathbb{R}^{X \times Y}$.
    \item $E_3$: Water Depth ($WD$). A 2D dynamic layer of dimension $X \times Y$. It represents the primary physical state of the cell 
    at time $t$, storing the surface water depth in meters: 
    $E_3 \subset \mathbb{R}^{X \times Y}$.
    \item $E_4$: Rainfall ($RF$). A 3D dynamic layer of dimension $X \times Y \times T$, explicitly incorporating the temporal 
    domain to inject discrete precipitation volumes ($\Delta V_{in}$) into the cell column during each time step $\Delta t$: 
    $E_4 \subset \mathbb{R}^{X \times Y \times T}$.
    \item $E_5$: Water Movement State ($WMS$). A 2D dynamic logical layer of dimension $X \times Y$ storing discrete states ($\text{int8}$). 
    It tracks whether a cell is hydraulically active (Dynamic vs. Static). This layer acts as a structural optimization shield, allowing 
    the execution engine to skip sub-threshold cells via dynamic bounding boxes: 
    $E_5 \subset \mathbb{N}^{X \times Y}$.
    \item $E_6$: Flood Risk Level ($FR$). A 2D dynamic categorical output layer of dimension $X \times Y$. It classifies the hazard level 
    of each cell at time $t$ by evaluating a heuristic threshold function over the current water depth ($WD \xrightarrow{g} FR$), mapping 
    physical outputs into discrete risk levels for real-time hazard mapping: 
    $E_6 \subset \mathbb{N}^{X \times Y}$.
\end{enumerate}


\subsubsection{Vicinity Function ($V$) and Nucleus Function ($N$)}

The \textbf{Vicinity Function} ($V$) determines the subset of adjacent cells that influence the calculation of the future state of a 
given cell. A Moore neighborhood with radius $r=1$ is adopted, defining a set of 8 adjacent cells connected orthogonally and 
diagonally.

$$V(i,j) = \{ (i+u, j+v) \in R \mid \max(|u|, |v|) \le 1 \}$$

The \textbf{Nucleus Function} ($N$) is projected onto the central cell itself to update the dynamic states of the system ($E_3$, $E_5$, 
and $E_6$), mapping the spatio-temporal modifications through local evolution rules: $N(i,j) = \{ (i,j) \}$.


\subsubsection{Intra-layer Transitions: Horizontal Propagation Dynamics}

Intra-layer transitions define the kinematic behavior of surface water flow across the discretized spatial grid. While inter-layer 
transitions govern vertical fluxes (such as precipitation and infiltration), this section focuses on horizontal propagation, where the 
state of a cell at time $t+1$ is determined by the hydraulic interactions with its immediate neighborhood at time $t$.

\paragraph{A. Neighborhood Formalization and Hydraulic Gradients}

Following the technical implementation in C++ and TensorFlow, the system employs a Moore neighborhood ($k=8$). This choice allows for 
a more realistic flow propagation that is significantly less dependent on grid orientation compared to the simpler von Neumann 
neighborhood.

For every cell $c_{i,j}$, the simulation engine computes a vector of hydraulic gradients $S$ toward its eight neighbors. The gradient 
is defined by the difference in potential energy (terrain elevation $H$ plus water depth $WD^*$) divided by the Euclidean distance $L_{k}$:

$$S_{k} = \frac{(H_{i,j} + WD^*_{i,j}) - (H_{neighbor,k} + WD^*_{neighbor,k})}{L_{k}}$$

Where $L_{k}$ corresponds to the cell size for cardinal neighbors and $\sqrt{2} \times \text{cell\_size}$ for diagonal neighbors. 
Adhering to the physical logic of the model, only positive gradients ($S_k > 0$) are considered, ensuring that water only moves toward 
cells with lower hydraulic potential.

\paragraph{B. Velocity Estimation via Manning’s Equation}

To optimize computational performance and ensure numerical stability, the model identifies the maximum local hydraulic gradient 
$S_{max} = \max(S_k)$ from the set of energy slopes toward the eight adjacent neighbors in the Moore neighborhood.

To account for varying fluid rheologies, the multi-physics core dynamically evaluates two distinct physical regimes—turbulent and 
laminar—computing a theoretical velocity for each:

\begin{enumerate}
    \item \textbf{Turbulent Regime (Clear Water Flow)}: The model integrates surface friction through a roughness coefficient $n$ 
    (Manning’s $n$), which is directly retrieved from the persistent Surface Roughness layer ($E_2$) at the target coordinates. 
    Approximating the hydraulic radius by the water depth ($WD^*$), the turbulent velocity ($v_{turb}$) is derived from a tensor-optimized 
    version of Manning’s Equation:

    $$v_{turb} = \frac{1}{n} \cdot {WD^*}^{2/3} \cdot \sqrt{S_{max}}$$

    \item \textbf{Laminar / Viscous Regime (Dense, Mud/Non-Newtonian Flow)}: To accurately simulate hyper-concentrated sediment or 
    mudflows, a viscous velocity ($v_{lam}$) is computed. This formulation incorporates fluid density ($\rho$), gravitational 
    acceleration ($g$), and dynamic fluid viscosity ($\mu$):

    $$v_{lam} = \frac{\rho \cdot g \cdot {WD^*}^2 \cdot S_{max}}{3\mu}$$
\end{enumerate}

The multi-physics core determines the final effective velocity ($v_{approx}$) by selecting the minimum value between the two 
regimes. This ensures the flow is naturally bounded by the dominant physical constraint at any given cell:

$$v_{approx} = \min(v_{turb}, v_{lam})$$

This dynamic velocity approximation represents the actual potential speed of the flood wave under highly variable rheological 
conditions—a capability critical for capturing the complex dynamics of events like the 2024 Valencia DANA. The resulting 
$v_{approx}$ then serves as the foundation for all subsequent volumetric displacement and multi-directional routing calculations.

\paragraph{C. Volumetric Displacement Calculation}

Once the effective velocity is determined, the engine must translate this linear speed ($m/s$) into a physical volume of water ($m^3$) 
capable of moving across cell boundaries. This is achieved by first calculating the Cross-Sectional Area ($A_{cs}$) of the flow, which, 
in our raster-based discretization, is defined as the product of the current water depth and the cell resolution:

$$A_{cs} = WD^* \cdot \Delta x$$

Subsequently, the Theoretical Outflow Volume ($V_{theor}$) is derived using the displacement equation over the simulation time step 
$\Delta t$:

$$V_{theor} = v_{approx} \cdot A_{cs} \cdot \Delta t$$

This volumetric representation is the cornerstone of the model's mass conservation strategy. By computing $V_{theor}$ before the routing 
phase, the engine ensures that the subsequent distribution of water to neighbors is based on finite physical quantities, preventing the 
numerical "inflation" of water levels often found in simpler CA models.

\paragraph{D. Multiple Flow Direction (MFD) and Mass Balance}

In contrast to traditional Single Flow Direction (D8) models, this simulator implements a Multiple Flow Direction (MFD) algorithm. The 
total outflow volume from a cell is distributed among all neighbors with a lower potential, weighting the amount of water sent to each 
neighbor according to the magnitude of their respective gradients.

The final mass balance for the surface water layer is governed by the following discrete differential equation:

$$WD_{i,j}^{t+1} = {WD^*}_{i,j}^{t} + \left( \frac{\sum \Delta V_{in} - \sum \Delta V_{out}}{Area_{cell}} \right)$$

To ensure numerical stability and prevent artificial oscillations (particularly in areas with steep slopes), the implementation includes 
a precision filter ($\epsilon$) and a flow-limiting constraint: the total outflow can never exceed the volume of water physically 
available in the cell during the current time step.

\paragraph{E. Execution Shielding via Water Movement State}

To maximize computational throughput, the evolution of the Water Movement State layer ($E_5$) acts as a dynamic gatekeeper. At the end 
of each time step $t$, the state of a cell transitions between static and dynamic regimes based on a dry-tolerance threshold 
($\delta_{dry}$) and neighborhood activity:

$$WMS_{i,j}^{t+1} = \begin{cases} 
\text{Dynamic}, & \text{if } \sum \Delta V_{out} > 0 \lor \sum \Delta V_{in} > 0 \\ 
\text{Static}, & \text{otherwise} 
\end{cases}$$

Cells flagged as static are completely bypassed by the C++ execution engine in the subsequent iteration, preventing redundant tensor 
operations in dry areas of the river basin.

\paragraph{F. Real-Time Hazard Categorization}

The final phase of the intra-layer loop updates the Flood Risk Level layer ($E_6$). This categorical state is determined via a discrete 
transfer function $g$ applied to the newly computed water depth, mapping continuous physical dimensions into emergency management 
indicators:

$$FR_{i,j}^{t+1} = g(WD_{i,j}^{t+1})$$

Where $g$ maps the depth to predefined risk intervals (e.g., Low, Medium, High, Severe Risk), ensuring that the cellular space outputs 
action-ready spatial data at every interval $\Delta t$.

\subsubsection{Inter-layer Transitions: Vertical Propagation Dynamics}

Inter-layer transitions govern the vertical interactions between the different information layers within a single cell $c_{i,j}$. 
Unlike horizontal propagation, these processes are local to the cell's vertical column and represent the exchange of mass and 
properties between the atmospheric, surface, and sub-surface layers.

\paragraph{A. Precipitation Injection (Atmosphere-to-Surface)}

The primary driver of the simulation, particularly in the context of the Valencia DANA, is the localized and intense rainfall. In the 
$m:n-CA^k$ framework, precipitation is treated as a dynamic input layer $P$ that contributes directly to the water depth layer ($WD$).

For each simulation step $\Delta t$, the water depth is updated to account for the incoming volume:

$${WD^*}_{i,j}^{t} = WD_{i,j}^{t} + \left( R_{i,j}^{t} \cdot \Delta t \right)$$

Where $R_{i,j}^{t}$ represents the rainfall intensity ($mm/h$ converted to $m/s$) at that specific coordinate.

\paragraph{B. Land Cover to Surface Roughness Generation}

A critical inter-layer process in the proposed model is the transformation of discrete geospatial data into continuous physical parameters 
during the initialization phase ($t=0$). The external Land Cover layer acts as a descriptor for the surface properties, which is mapped 
onto the persistent Surface Roughness layer ($E_2$) via a Lookup Table (LUT):

$$\forall c_{i,j} \in R \implies E_{2(i,j)} = \text{LUT}(\mathcal{L}_{LandCover}(i,j))$$

This structural transition ensures that once the simulation loop starts, the raw Land Cover layer can be purged from memory. When the 
intra-layer propagation logic (Manning’s Equation) is executed, it directly queries $E_2$, capturing the hydraulic complexity of the 
Valencian metropolitan area by recognizing that water moves slower through the dense urban streets of Paiporta compared to the smooth 
paved surfaces of the V-31 highway.



\section{Software Architecture: The Hexagonal Approach}
\label{sec:hexagonal-methodology}

One big problem in computing is that the physical models are really closely tied to the underlying infrastructure. This includes things 
like the formats we use to store data, the ways that computers talk to each other and the special libraries that are made for types of 
hardware. In a lot of hydraulic simulators the rules for how things change are all mixed up with the code that reads specific types of 
files like GeoTIFF or CSV files. This makes it really hard to keep the system running test it or make changes to it.

To get around these problems, this research uses the Hexagonal Architecture or the Ports and Adapters pattern. The main goal is to keep 
the Simulation Core separate from other things that might affect it. By making a line between the "domain logic”, which is just the 
physics of how floods work; and the "infrastructure”, which is how we get and send data. The simulator becomes a highly flexible and 
technologically agnostic engine.

This way of designing the system is especially important, for a Reactive Digital Twin. It lets the system switch from reading data from 
HDF5 files to getting real-time data from MQTT rainfall streams without changing any of the code that controls how the hydraulic system 
changes. The hydraulic Simulation Core stays the same.


\subsection{Core vs. Infrastructure}

The Hexagonal design divides the system into two distinct regions: the Inside (The Core) and the Outside (The Infrastructure). The 
fundamental constraint governing this architecture is the Dependency Rule, which states that dependencies must always point inwards 
toward the Core.

\paragraph{A. The Core (The "Inside")}

The Core represents the "Hexagon" itself. It contains the pure business and scientific logic of the project. In this thesis, the Core 
is composed of:

\begin{itemize}
    \item \textbf{The $m:n-CA^k$ Engine}: The implementation of the transition functions (Manning’s equation, MFD algorithm) as defined in Section 3.1.

    \item \textbf{The Domain Model}: The tensorial representation of the grid, cell states, and hydraulic parameters.

    \item \textbf{Application Services}: Orchestrators that handle the simulation loop and time-step synchronization.
\end{itemize}

The Core has no knowledge of the outside world. It does not know if the data comes from a local disk or a cloud-based broker; it simply 
defines the interfaces it needs to operate.

\paragraph{B. The Infrastructure (The "Outside")}

The Infrastructure consists of everything that interacts with the Core. It is responsible for the "technical details" that the 
Core ignores. This includes:

\begin{itemize}
    \item \textbf{Data Sources}: File I/O drivers, spatial data parsers, and static storage handlers.

    \item \textbf{State Computation}: Hardware-accelerated execution engines for tensor-based matrix operations.

    \item \textbf{Communication}: Network protocols for real-time telemetry and data serialization mechanisms for external visualization.
\end{itemize}

By segregating these concerns, we ensure that technological obsolescence in the infrastructure (e.g., a protocol update or a 
change in GIS standards) does not compromise the scientific validity of the simulation engine.


\subsection{Technical Advantage: Testability and Mocking}

A significant advantage of this specification is the ability to perform Unit Testing on the hydraulic logic. Because the Core is 
decoupled from the hardware and files, we can create "Mock Adapters" that feed the Core with pre-defined, small-scale tensors to 
verify the mathematical correctness of the $m:n-CA^k$ transitions without needing a full GIS environment or an active MQTT broker.



\section{Distributed Communication Model (MQTT)}
\label{sec:distributed_communication_model}

The operational effectiveness of a real-time pluvial flood emergency system depends not only on the mathematical precision of its 
calculation engine but also on the latency, throughput, and reliability of its data delivery infrastructure. To bridge the gap 
between the high-performance computational core and remote visualization or decision-support clients, the analytical framework 
incorporates a distributed telemetry layer governed by the Message Queuing Telemetry Transport (MQTT) protocol. This lightweight, 
publish-subscribe, machine-to-machine (M2M) communication standard is explicitly selected to facilitate low-overhead, asynchronous 
data dissemination over potentially unstable networks, matching the strict telemetry requirements dictated during extreme weather 
crises such as the 2024 Valencia DANA event.


\subsection{Decoupling Strategy: Publisher-Subscriber Paradigm}
\label{subsec:decoupling_strategy}

To guarantee that the rigorous computational cycles of the core simulation engine are never throttled or bottlenecked by slower 
external network loops or hardware-constrained rendering clients, the communication layer relies strictly on the Publisher-Subscriber 
pattern. This pattern enforces an absolute architectural separation between the simulation engine (the "Model") and the remote 
monitoring applications (the "View"). This structural decoupling operates across three primary dimensions:

\begin{itemize}
    \item \textbf{Spatial and Identity Anonymity}: The simulation core functions as an isolated data publisher, remaining completely 
    agnostic of the network topology, geographical locations, or total count of downstream consumers. This allows multiple heterogeneous 
    entities—ranging from municipal emergency centers to regional civil protection dashboards—to dynamically attach to the live 
    telemetry stream concurrently without imposing any additional memory or computational burden on the core engine execution thread.
    
    \item \textbf{Temporal Decoupling}: The computational execution of hydraulic time steps remains entirely independent of the 
    consumption or ingestion rates of remote clients. If a network disturbance occurs or if a visualization system experiences 
    local rendering lag, the intermediate message broker acts as an independent buffer. This isolates the simulator from client 
    performance drops and preserves real-time simulation guarantees.
    
    \item \textbf{Functional Isolation}: By abstracting network interfaces into pure outward-facing data ports, the physical and 
    mathematical formalization of cell state transitions remains completely unpolluted by protocol-specific constraints, transport 
    framing routines, or session management policies.
\end{itemize}


\subsection{Formal State Distribution Channels and Topic Hierarchy}
\label{subsec:topic_hierarchy}

To formalize the interaction between the discrete simulation space and external virtual or visualization environments, the communication 
framework maps core state transitions into structured, event-driven telemetry channels. This systematic classification draws directly 
upon the distributed simulation paradigms established by \cite{FonsecaICasas2013}, where a clear separation is maintained between behavioral 
model execution, experimental parameters, and the graphical representation of entities across distinct logical pathways. Under this formal 
language framework, simulation events are formalized into explicit, independent signals routed through a strictly hierarchical, isolated 
namespace.

To support concurrent multi-scenario emergency evaluations within a unified distributed cluster, the hierarchy roots its namespace in a 
unique scenario identifier. Information is then segregated into three functional categories based on its operational lifecycle and 
transmission frequency:

\begin{itemize}
    \item \textbf{Lifecycle and Peer Discovery Channel} (\texttt{FloodSim/\{scenarioName\}/system/handshake}): This low-frequency 
    administrative channel manages the operational readiness of the distributed node network. It is exclusively reserved for initial 
    peer handshakes, entity discovery, client capability negotiations, and mutual connection verification before simulation initialization.
    
    \item \textbf{Structural and Physical State Broadcast Channel} (\texttt{FloodSim/\{scenarioName\}/events}): This high-traffic channel 
    serves as the primary data pipeline for the system. It handles a dual-phase transmission schema: (i) an initial, structural setup 
    phase that broadcasts the domain's geographical boundaries, cell resolutions, and spatial matrix frameworks; and (ii) a continuous 
    execution phase that broadcasts the dynamic, high-velocity distribution of water depths and computed hazard classification levels 
    across the simulation grid.
    
    \item \textbf{Bi-directional Control and Orchestration Channel} (\texttt{FloodSim/\{scenarioName\}/control/} \\ \texttt{events}): This channel 
    provides an out-of-band feedback loop from the consumers back to the simulation coordinator. It handles upstream control commands, 
    backpressure metrics, and frame-level reception acknowledgments, enabling active flow control between distributed computing nodes.
\end{itemize}

\subsection{Differential State Framing and Temporal Synchronization Protocol}
\label{subsec:temporal_synchronization}

Disseminating high-resolution simulation grids across a distributed network introduces significant bandwidth and state-coherency 
challenges. To address these limitations theoretically, the methodology replaces full-state data sweeps with a 
\textit{Differential State Update} philosophy. 

Instead of broadcasting the entire cellular matrix during each time step—which would saturate network capacity—the system isolates and 
transmits only the delta changes (i.e., cells whose hydraulic properties have drifted beyond a predefined physical tolerance). To 
maintain an helical, chronologically coherent state machine across remote visualizers, the telemetry stream follows a strict four-phase 
orchestration sequence for every simulation step $t$:

\begin{enumerate}
    \item \textbf{Frame Initialization Phase}: An outbound control boundary signal is emitted to declare the onset of a new temporal 
    frame, specifying the total volume of discrete data sub-packets that constitute the upcoming state update.
    
    \item \textbf{State Modification Phase}: The actual physical updates are dispersed across the network in decoupled, bounded streams, 
    carrying the differential cell coordinates, localized fluid depths, and risk levels.
    
    \item \textbf{Frame Termination Phase}: A boundary closure signal is transmitted, notifying the remote visualization engine that all 
    differential updates for time step $t$ have been successfully delivered and that the local scene graph can be safely refreshed.
    
    \item \textbf{Temporal Synchronization Phase}: A final synchronization heartbeat maps the internal simulated execution clock to the 
    external rendering time base. This operational sequence prevents visual artifacts, such as partial frame rendering or temporal 
    drifting, ensuring that all distributed nodes maintain a logically consistent and identical view of the unfolding disaster.
\end{enumerate}


\section{Data Engineering and Tensor Representation}

The efficiency of a Cellular Automata simulation is fundamentally constrained by its data access patterns. While traditional GIS 
software operates on various file formats and coordinate systems, a high-performance simulator requires a unified, contiguous, and 
multidimensional memory layout. This section details the "Data-as-Interface" philosophy, where geospatial information is abstracted 
into tensors to enable vectorized computation.


\subsection{The Hierarchical Spatiotemporal Framework}
\label{subsec:hierarchical_spatiotemporal_framework}

To effectively manage dynamic environmental boundaries—such as high-velocity, time-varying pluvial rainfall fields—without introducing 
the architectural overhead and heavy runtime dependencies of complex binary database containers (e.g., HDF5 or NetCDF), the analytical 
engine establishes a customized, decoupled storage convention termed the Hierarchical IDRISI Format (HIF). This format is conceived as 
a structured, lightweight specification designed to emulate three-dimensional (3D) spatiotemporal data cubes using native 
two-dimensional (2D) GIS raster matrices as its fundamental computational building blocks.

\subsubsection{Conceptual Abstraction and Volumetric Representation}
\label{subsub:conceptual_abstraction}

From a conceptual perspective, the framework characterizes continuous temporal phenomena as a discrete chronological sequence of 
independent spatial states. Let a continuous spatiotemporal variable $\mathcal{F}(x, y, t)$ be discretized over a spatial grid of 
dimensions $N_x \times N_y$ and a total simulated duration divided into $T$ discrete intervals. The HIF architecture represents this 
3D data volume not as a monolithic binary block, but as an ordered set of isolated 2D tensor slices:

\begin{equation}
    \mathcal{F} = \{ \mathbf{M}_1, \mathbf{M}_2, \dots, \mathbf{M}_t, \dots, \mathbf{M}_T \}
\end{equation}

where each matrix $\mathbf{M}_t \in \mathbb{R}^{N_x \times N_y}$ encapsulates the spatial distribution of the environmental variable at 
the exact discrete time-step $t$.

This structural formalization directly enables a \textit{lazy loading} paradigm within the execution loop. Instead of allocating massive, 
contiguous memory buffers to hold the entire lifetime of the historical or predictive dataset prior to simulation initialization, the core 
execution engine isolates and ingests only the specific spatial frame $\mathbf{M}_t$ strictly required for the active hydraulic time-step 
computation. Consequently, the memory footprint of the simulator scales independently of the total duration ($T$) of the meteorological 
event, satisfying the strict hardware-boundary constraints dictated by real-time crisis deployment scenarios.

\subsubsection{Components of the Spatiotemporal Dataset}

The framework conceptualizes high-dimensional temporal datasets through three primary logical layers:

\begin{enumerate}
    \item \textbf{Global Spatial Descriptor}: A master metadata element defining the unified coordinate reference system, spatial 
    boundaries, and spatial resolution.

    \item \textbf{Temporal Attribute Manifest}: An orchestrated index mapping discrete simulation time-steps to structural metadata, 
    containing temporal downscaling factors and chronological timestamps.

    \item \textbf{Temporal Frame Repository}: A contiguous collection of localized raw matrix snapshots, where each element encapsulates 
    the spatial state variables of a single independent time-step.
\end{enumerate}

\subsubsection{Chronological Event Synchronization and Spatial Downgrading}
\label{subsub:chronological_synchronization}

Environmental boundary conditions within real-time pluvial flood dynamics present heterogeneous temporal lifecycles. While physical domain 
attributes remain strictly static, boundary forcing fields such as precipitation exhibit dynamic, non-uniform state changes. To align these 
scales without enforcing rigid, fixed-step simulation intervals, the framework establishes a decoupling mechanism governed by a 
timestamp-driven chronological validation loop.

Let the active state of the simulation clock be represented as a continuous time point $t \in \mathcal{T}$. A dynamic environmental layer 
$\mathcal{F}$ maps its state transitions to an ordered vector of explicit historical or predictive timestamps:

\begin{equation}
    \mathcal{T}_{\text{layers}} = \{ \tau_0, \tau_1, \dots, \tau_k, \dots, \tau_K \}
\end{equation}

The execution engine evaluates the temporal validity window at every integration increment. An ingestion event is triggered to advance the 
current spatial configuration from frame $k$ to $k+1$ if and only if the active simulation time passes the next chronological threshold:

\begin{equation}
    t \ge \tau_{k+1}
\end{equation}

Furthermore, environmental forcing inputs frequently operate at a coarser spatial resolution than the high-fidelity computational grid 
designated for hydraulic routing. To bridge this spatial gap without generating runtime memory allocation overhead, the methodology 
incorporates a programmatic \textsl{spatial downgrading mapping}. 

Let a cell in the fine-resolution simulation grid be indexed by its matrix coordinates $(r, c)$, and let the coarse input dataset possess a 
spatial resolution scaled down by an integer factor $\gamma$. The framework defines a discrete spatial projection mapping 
$\mathcal{P}: \mathbb{N}^2 \rightarrow \mathbb{N}^2$ such that:

\begin{equation}
    \mathcal{P}(r, c) = \left( \lfloor r / \gamma \rfloor, \, \lfloor c / \gamma \rfloor \right)
\end{equation}

Whenever a temporal frame update is approved, the system executes a direct cell-by-cell coordinate projection. If $\gamma = 1$, a 
direct 1:1 state transformation is applied. If $\gamma > 1$, the engine streams the coarse spatial matrix into an intermediate resampled 
buffer before mapping the forcing coefficients onto the active high-resolution simulation tensor, ensuring deterministic spatial alignment.


\subsection{The GIS-to-Tensor Pipeline: From Rasters to Multi-Channel Cubes}

The transition from environmental raw data to a simulation-ready state follows a specialized Extract, Transform, Load (ETL) framework. 
This process ensures that diverse environmental layers—specifically topography, land cover, and rainfall—are spatially and temporally 
synchronized to establish a mathematically consistent initial state for the cellular automata model.

\paragraph{A. Spatial Alignment and Domain Standardization}

Raw environmental datasets typically exhibit heterogeneous spatial resolutions, diverse geometric extents, and distinct geographic 
projections. To ensure spatial consistency across the simulation domain, the pipeline executes two primary geometric transformations:

\begin{itemize}
    \item \textbf{Coordinate Reference System (CRS) Standardization}: All geospatial layers are projected to a common metric coordinate 
    system (UTM 30N). This uniformity is mathematically required to ensure that Euclidean distance calculations within the Moore 
    neighborhood remain valid and isotropic across the spatial grid.

    \item \textbf{Resampling and Spatial Discretization}: Spatial layers are resampled to a uniform grid extent of dimensions $H \times W$. 
    The pipeline employs bilinear interpolation for continuous environmental variables (such as elevation models) and nearest-neighbor 
    interpolation for discrete thematic categories (such as land use maps) to preserve categorical boundaries.
\end{itemize}

\paragraph{B. Tensorization: The Multi-Channel Spatial Representation}

Once spatially aligned, the heterogeneous physical layers are mapped into a unified algebraic structure. The localized state of the river 
basin is represented as a 3D tensor of shape $(C, H, W)$, defined as follows:

\begin{itemize}
    \item \textbf{Channels ($C$)}: Represents the $n$ distinct physical and environmental information layers dictated by the $m:n-CA^k$ 
    model architecture (e.g., Channel 0: Water Depth, Channel 1: Land Use, Channel 2: Topography).

    \item \textbf{Height and Width ($H, W$)}: Represent the dimensions of the discretized spatial grid.
\end{itemize}

By structuring the spatial domain as a multi-channel tensor, the geographical space is treated as a single mathematical object. This 
formulation allows the cellular automata transition rules to be expressed as global tensor operations and matrix shifts, shifting the 
computational paradigm from local, independent cell evaluations to unified spatial transformations.


\subsection{Data Pipeline Design and Pre-processing Framework}

The core of the flood simulation system relies on its ability to process vast amounts of heterogeneous geospatial data accurately and 
efficiently. The data pipeline is conceived as a centralized conceptual orchestrator responsible for the ingestion, transformation, and 
provisioning of the physical parameters required by the hydraulic model.

The primary methodological objective is to decouple data acquisition from the underlying simulation logic. By isolating the data ingest 
mechanics, the system guarantees that variations in external data sources or file formats do not compromise the mathematical stability of 
the simulation engine. This framework is structured upon three fundamental pillars:

\begin{enumerate}
    \item \textbf{Spatial Integrity}: Ensuring that spatial coordinate reference systems, geometric extents, and resolutions remain 
    mathematically consistent across all processed data layers.

    \item \textbf{Scalability}: Allowing the seamless inclusion of new physical variables (e.g., soil moisture, infiltration capacity) 
    without requiring structural changes to the core simulation flow.

    \item \textbf{Automation}: Minimizing manual intervention during the preprocessing stages of raw geospatial datasets to ensure workflow 
    reproducibility.
\end{enumerate}

\subsubsection{Data Categorization: Static vs. Dynamic Layers}

To optimize domain state updates and operational memory management, the methodology categorizes geospatial information into two distinct 
functional types based on their temporal dynamics during a simulation event:

\begin{itemize}
    \item \textbf{Static Layers}: These represent environmental characteristics that remain invariant throughout the simulated event. 
    Examples include the Digital Elevation Model (DEM) and Land Use. These layers are processed a single time during 
    initialization and serve as the immutable topographic canvas of the simulation domain.

    \item \textbf{Dynamic Layers}: These comprise time-varying boundary conditions and state variables that actively drive the transition 
    functions of the model. Examples include spatialized Rainfall Intensity. The methodology must formalize a sequential state retrieval 
    mechanism, ensuring that the simulation engine receives the exact temporal "snapshot" corresponding to each discrete time step.
\end{itemize}

This conceptual dichotomy allows the pipeline to separate the operational processing paths for each layer type, maximizing data-flow 
efficiency before execution.

\subsubsection{Data Flow Logic and Validation Principles}

The progression of data through the preprocessing pipeline is governed by a deterministic lifecycle, ensuring that the simulation engine 
receives a spatially and temporally synchronized state of the environment.

\paragraph{A. Configuration-Driven Parameterization}

The pipeline initializes via a centralized metadata control plane. This approach separates the layer definitions—which are structurally 
bound to the pipeline's logic for consistency—from their operational parameters (e.g., geographic boundaries, spatial resolutions, and 
specific interpolation models). This externalization ensures scientific reproducibility, enabling identical pipeline logic to be deployed 
across distinct geographic catchments by adjusting numerical and spatial constants.

The control plane orchestrates three main dimensions:

\begin{enumerate}
    \item \textbf{Global Simulation Metadata}: Defines temporal boundaries (start/end timestamps) and domain constraints to synchronize 
    all incoming layers within a uniform space-time window.

    \item \textbf{Layer Parameterization}: Dispatches execution parameters to individual layers, including target resolutions, spatial 
    boundaries, and interpolation models (such as variogram specifications for rainfall spatialization).

    \item \textbf{Visualization Diagnostics}: Identifies which physical layers require graphical rendering for research and diagnostics, 
    isolating them from background computational layers.
\end{enumerate}

\paragraph{B. Spatial Synchronization and Dependency Mapping}

Before executing any raster transformations, the pipeline must validate the logical structure of the data dependencies and enforce spatial 
convergence. This validation is modeled using Graph Theory principles:

\begin{itemize}
    \item \textbf{Directed Acyclic Graph (DAG) Resolution}: The operational dependencies between physical layers (e.g., a rainfall layer 
    requiring an elevation model for spatial clipping) are modeled as nodes and directed edges. The pipeline enforces a topological sort 
    to guarantee that circular dependencies are nonexistent and that each component is initialized only when its structural prerequisites 
    are met.

    \item \textbf{Master Grid Alignment}: To mitigate spatial misalignment errors to which cellular automata are highly sensitive, the 
    framework enforces a "Master Grid" paradigm. The primary topographic layer (DEM) establishes the baseline spatial constraints. All 
    secondary layers (e.g., roughness, rainfall) are validated, resampled, and clipped against this reference to guarantee exact overlap 
    in Coordinate Reference System (CRS), pixel resolution, and bounding boxes.
\end{itemize}

\paragraph{C. Sequential Execution}

The pipeline adheres to a strictly sequential execution model, processing each geospatial layer in a deterministic order derived from the 
topological sorting. Each environmental variable is isolated within its own functional generator, encapsulating the transformation rules
to preserve strict modular boundaries.
